sc_read_kallisto_wrapper

Summary

Takes as input a BCL file generated by an Illumina sequencer run on a 10XGENOMICS Chromium single cell 3' RNA-seq library, extracts the reads, maps them to a transcriptome using kallisto and formats the equivalence class counts as input for Seurat.

Description

The sc_read_kallisto_wrapper performs the following steps :
  1. Uncompress and detar the BCL archive.
  2. Read the RunInfo.xml and RunParameters.xml file from the BCL archive in order to determine what chemistry (SC3Pv1 or SC3Pv2) was used.
  3. Use Cell Ranger mkfastq (which itself relies on bcl2fastq) to extract sequences in fastQ format from the BaseCalls in the BCL archive. Cell Ranger sorts the sequences according to sample, using the sample barcodes. It allows one mismatch. Sequences that have a sample barcode different in more than 2 postions from any of the sample barcodes referenced in the CSV file are considered "Undetermined".
  4. Use python scripts to split up the sequences per cell relying on the cell barcodes. The scripts do their best to correct the sequencing errors that the cell barcodes can contain ; for this they need an estimate of how many cells there were in the sample and how many bases difference there is at least between any pair of cell barcodes.
  5. Use kallisto to map the reads to a transcriptome. Kallisto uses an index with all the k-mers in the transcriptome and creates a De Bruijn graph. It must handle the cases where a read can map at multiple locations as well as the cases where a read can map to alternative transcripts of the same gene. It does this by counting mappings of reads to equivalence classes. While working on single cell data, kallisto does not perform the second step of its algorithm, which involves assigning reads to individual transcripts, but it does use the UMI's (unique molecular identifiers) to reduce the bias caused by duplicates.
  6. Modify the kallisto output so that it becomes suited as input for Seurat and replace the equivalence class numbers in the kallisto output by transcript names.

Input files

The sc_read_kallisto_wrapper takes as input a BCL (BaseCalls) file from an Illumina sequencer. It must be a single file in archived (*.tar.gz) format ; the script will untar it on-the-fly.

The script also needs a CSV (comma separated format) file with information about which sequences to extract. You can use the IEM (Illumina Experiment Manager) file used with Illumina sequencers, but it can be more convenient to use the short format CSV file instead (see below).

The sc_read_kallisto_wrapper can optionally also take as input a kallisto index (for mapping the reads using kallisto) and a transcriptome file (for converting the numbers in the kallisto output to transcript names). The current version of the sc_read_kallisto_wrapper only supports a transcriptome given as a single uncompressed file in fastA format. Note that it is important that the transcriptome file is the same that was used to make the kallisto index since the script relies on the order of the sequences in the file. Note that, since you can only provide one kallisto index, if the Illumina flowcell contained samples from different organisms, you must either process the BCL file in separate sc_read_kallisto_wrapper runs or alternatively use a Kallisto index made from a mixed transcriptome.

short CSV file format

The CSV file must start with a header line : 
Lane,Sample,Index
The header must be followed by a series of lines of type : 
1,test_sample,SI-P03-C9
where :
  • 1 is the lane numer (can only take values 1, 2, 3, 4)
  • test_sample is a sample name, which can be chosen arbitrarily
  • SI-P03-C9 is the identifier of the set of sample barcodes used to identify the sample. You can find what the acceptable codes are by typing a letter in the Index Set: box of the Sample Sheet Generator on the Cell Ranger Web site.
Note that if a sample has been spread over different lanes you will have to write lines with different lane number but same sample name and barcode identifier. Do not include the lane number in the sample name !

Output files

The sc_read_kallisto_wrapper produces a series of files in gzipped fastQ format. These files are grouped in a folder called FASTQ_FILES and have names of type :

test_sample_S1_L001_R1_001.fastq.gz

where :

  • test_sample is the sample name. Reads that could not be assigned to a sample, either because their barcode was not sequenced properly or because their barcode set identifier was not mentioned in the CSV file, are put in files with name Undetermined.
  • S1 is a sample number that is incremented according to order in the CSV file. Undetermined sequences get number S0.
  • L001 is the lane number.
  • R1 is a read identifier. Identifier L1 stands for the Illumina sample barcodes (8 bases long). In case the older CHROMIUM10X chemistry SC3Pv1 was used R1 , R2 and R3 stand respectively for the sequence, the cell barcode (14 bases) and the UMI (10 bases). With the newer chemistry SC3Pv2 there is only R1 with the cell barcode (16 bases) and the UMI (10 bases) as one sequence and R2 with the actual sequence.
  • 001 is always 001.
There is also a file qc_summary.json with quality control metrics (see below).

If a kallisto index is provided the sc_read_kallisto_wrapper will for each sample create a folder <sample_name>_kallisto with the raw kallisto output, as well as a folder <sample_name>_4Seurat with input suited for Seurat.

The kallisto output has 3 files :

  • matrix.cells : a list of unique identifiers for cells, including the cell barcode
  • matrix.ec : a two-column file in tab-separated format. The first column contains incrementing numbers, the second column contains numbers or groups of numbers that refer to the order of the transcripts in the transcript file used to make the kallisto index and correspond to equivalence classes.
  • matrix.tsv : a three-column file in tab-separated format. The first and the second column correspond to a line number in respectively matrix.ec and matrix.cells and the third column contains the mapped reads count for that cell-equivalence class pair.
Note that in kallisto output numbering starts at 0.

The Seurat input has 3 files :

  • barcodes.tsv : idem as matrix.cells
  • genes.tsv : idem as matrix.ec, with numbering starting at 1. If a transcriptome is provided the transcript numbers are replaced by transcript names.
  • matrix.mtx : idem as matrix.tsv, with numbering starting at 1.
It will also make a graphic <sample_name>_umi_barcodes.png with the number of UMI found (before the attempt to correct them). The height of the graph indicates the number of UMI's attached to the same cell barcode ; the barcodes are sorted according to decreasing UMI count. The cut-off used to select cells is indicated by a green dashed line.

Quality control metrics

The file qc_summary.json contains information that can be useful to assess whether the sequencing went fine. It contains a list of metrics for each sample. The most important are :
barcode_exact_match_ratioThe percentage of barcode sequences that exactly match a whitelisted 10X barcode.
barcode_q30_base_ratioThe percentage of barcode bases at or above Q30.
bc_on_whitelistThe percentage of barcode sequences that match a 10X barcode on the whitelist, post error-correction. Corresponds to the "Valid Barcodes" value in cellranger output metrics.
mean_barcode_qscoreMean quality score of barcode bases.
number_readsReads per lane matching the sample's sample index (or overall in 'all')
read1_q30_base_ratioThe percentage of R1 bases at or above Q30.
read2_q30_base_ratioThe percentage of R2 bases at or above Q30.

Note

Determining just how many different cells there are in the sample, in the light of sequencing errors in the cell barcodes, is a problem for which there is no generally accepted solution. We use the following procedure : sort the cell barcodes according to decreasing number of corresponding UMI's, note how many UMI's correspond to the Nth cell, where N is 1/100 of the estimated number of cells, then accept all cells that have at least 1/10 of that number of UMI's.
